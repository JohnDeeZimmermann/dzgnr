# Requirements: Proper CMYK Support

## Goal

Dzgnr should produce print-oriented PDFs with true CMYK color output by default, rather than only Chromium-generated RGB PDFs with a warning. The implementation should preserve Dzgnr's current HTML-to-PDF authoring flow while adding a reliable post-render print conversion step and validation/reporting that can confirm the result is CMYK-oriented.

## User decisions captured

- Target: **true PDF CMYK** support.
- Workflow preference: decide based on research; user requested this be looked up.
- ICC profiles: use **bundled defaults** to keep the first workflow simple.
- Backward compatibility: **change the default** so print-ready/CMYK behavior becomes the normal Dzgnr output, even if output/dependencies change.

## Functional requirements

### FR1 — CMYK becomes the default output intent

Running the existing render command should attempt to produce CMYK-converted PDFs by default. Current Chromium-only output should no longer be reported as the successful final state when CMYK conversion is available.

### FR2 — Keep HTML/CSS design authoring

Users and AI agents should continue authoring designs as HTML/CSS and using existing size/page configuration. The conversion must happen after layout/rendering, not by replacing the design model.

### FR3 — Bundled ICC/default print profile

Dzgnr should include a documented default CMYK profile choice for first-class usage. Configuration may later expose profile selection, but the initial implementation should work without requiring users to download or pass profiles.

### FR4 — Post-render CMYK conversion pipeline

Dzgnr should render with Chromium/Playwright first, then convert the resulting PDF to CMYK with a toolchain that can emit real CMYK PDF content and embed/declare suitable output intent where practical.

### FR5 — Validation and reporting

Validation should report whether the final PDF appears to be CMYK/print-converted. The existing unconditional warning `Generated by Chromium; true CMYK/PDF-X output is not guaranteed.` should be replaced with more precise status messages.

### FR6 — Multi-page support must continue to work

CMYK support must handle both current output modes:

- `combined`: one multi-page PDF after merging/conversion.
- `separate`: one converted PDF per page.

### FR7 — Failure modes must be explicit

If the converter or bundled profile is unavailable or conversion fails, Dzgnr should fail clearly by default rather than silently returning RGB output as if it were print-ready.

### FR8 — Escape hatch for RGB/draft output

Because CMYK conversion adds dependencies and possible runtime constraints, the plan should include an explicit draft/RGB mode or configuration option for users who only need quick Chromium PDFs.

## Non-functional requirements

- Keep the CLI simple for AI agents: sane default, minimal new required flags.
- Prefer a robust, maintainable conversion toolchain over pretending browser CSS can guarantee CMYK.
- Preserve page dimensions, page count, and print backgrounds across conversion.
- Document installation/runtime requirements for any external binary dependency.
- Do not require writing tests in this planning phase; implementation should ask the testing subagent to create tests per project instruction.

## Open assumptions for research

- The exact converter should be chosen after online and codebase research.
- The exact bundled ICC profile must be legally redistributable or replaced by a documented install/download step if redistribution is not viable.
- PDF/X compliance is not the initial user-selected target, but the plan may recommend a path compatible with future PDF/X support.
